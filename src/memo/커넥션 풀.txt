개발자가 getConnection()을 통해 DB에 접근하려고 할 떄 아래와 같은 일이 발생한다. 
1. TCP/IP 연결
2. DB IP, username, password 전달
3. DB 내부에서 2의 정보를 바탕으로 인증 과정을 거치고 개발자만의 【DB 세션】 생성
4. 개발자에게 커넥션 반환
-> 그러나 만약 사용자가 동시에 100명이 오면, 동시에 위 과정을 100번 해야 하며, 이때마다 리소스 부하가 일어난다. 
이러한 문제점을 극복하고자 미리 몇 개의 커넥션을 만들어 놓고 요청될 떄마다 그냥 반환하는 커넥션 풀의 개념이 탄생한 것이다. 

커넥션 풀 숫자는 서비스의 특징와 애플리케이션 서버 스펙, DB 스펙에 따라 다르기 때문에 성능 테스트를 통해 최대 커넥션 풀 숫자를 설정해야 한다.
실무에서는 HICAKRI CP라는 커넥션 풀 오픈소스를 거의 대부분 쓴다.
스프링 부트 2.0부터는 HICARI CP를 디폴트로 등록해서 커넥션 풀을 사용한다.
