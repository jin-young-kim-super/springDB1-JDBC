아직 문제점이 있다
서비스 계층에 반복되는 코드가 여전히 잇다. 트랜잭션 매니저를 사용하더라도 각 메서드마다 트랜잭션을 실행을 할 떄 같은 코드를 여전히 반복해서 적여야 한다. 
이번 시간에는 트랜잭션 템플릿이라는 기능을 이용하여 해결할 것이다.
(핵심 개념은 템플릿 콜백 패턴인데, 고급편에서 자세히 다룰 예정이다, 그러니 지금은 자세히 이해를 못해도 ㄱㅊ)

아래 코드에서 this.transactionTemplate = new TransactionTemplate(transactionManager) 이런 코드가 있다.
근데 우리가 DIP 원칙을 생각하면 보통은 의존관계를 통하여 TransactionTemplate을 주입 받는 것이 정석이였다.
그러나 아래에서는 생성자에서 이미 TransactionTemplate을 주입을 받고 있다.
그 이유는 TransactionTemplate이 인터페이스가 아니고 구현 클래스라는 것에 있다.
DIP는 대개의 경우, OCP를 지키기 위하여 인터페이스를 선언하고 의존관계를 통해 주입을 받았다. 그러나 TransactionTemplate은
구현 클래스이다. 그래서 관례상 TransactionTemplate의 주입은 생성자에서 직접 해준다.
또한 생성자의 매개변수를 보면 트랜잭션 매니져가 있다. 이것은 관례상 예를 들어 JDBC 트랜잭션 매니저 구현체에서 JPA 트랜잭션 매니저 구현체로
변경이 됐을 때, 거기에 따른 코드 변경 없이 유연하게 TransactionTemplate에 트랜잭션 매니저 구현체를 주입하기 위함이다.


public class MemberServiceV3_2 {

    //private final PlatformTransactionManager transactionManager;
    private final TransactionTemplate transactionTemplate;
    private final MemberRepositoryV3 memberRepository;


    public MemberServiceV3_2(MemberRepositoryV3 memberRepository, PlatformTransactionManager transactionManager) {
        this.memberRepository = memberRepository;
        this.transactionTemplate = new TransactionTemplate(transactionManager);
    }
