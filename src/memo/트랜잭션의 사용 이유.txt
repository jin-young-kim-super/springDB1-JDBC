그냥 정보를 저장할 때, 데이터 베이스를 사용하지 않고, 파일에 저장하면 되지 왜 굳이 데이터 베이스를 사용하는 것일까?
여러 이유가 있겠지만, 가장 큰 이유는 데이터 베이스틑 트랜잭션 기능을 제공한다는 것이다. 

A가 B에게 1000원 계좌이체하는 과정을 생각해보자! 이떄 2개의 operation이 동작한다. 
1. A 계좌에서 -5000원
2. B 계좌에서 +5000원
-> 위 둘 중 1개라도 실패하면 치명적인 시스템 에러가 일어 난다. 
고로 둘 중 1개라도 실패하면 RollBack, 모두 다 성공하면 그때 Commit을 함으로써 데이터베이스의 ACID를 보장한다.

트랜잭션은 ACID를 보장(ACID는 트랜잭션이 보장하는 원칙이다)
1. 원자성(Atomic) : 트랜잭션 내에서 실행작 작업(operation)들은 【마치 하나의 작업】인 것처럼 모두 성공하거나 모두 실패하도록! 묶어줘야 한다.

2. 일관성(Consistency) : 모든 트랜잭션은 일광성 있는 데이터 베이스 상태를 유지해야 한다. 예를 들어 데이터베이스에서 정한 무결성 제약 조건(Contraint)을 항상 만족해야 한다.
-> 일관성(Consistency)은 트랜잭션 전후에 데이터베이스가 항상 유효한 상태를 유지하는 것을 의미합니다. 
즉, PK, FK, NOT NULL, CHECK 등의 제약 조건과 비즈니스 규칙을 만족하는 상태를 유지하는 것입니다. 트랜잭션 도중 오류가 발생하면 Rollback을 통해 이러한 일관성을 보장합니다.


3. 격리성(isolation) : 동시에 실ㅇ행되는 트랜잭션들이 서로에게 영향을 미치지 않도록 격리한다. 예를 들어 동시에 같은 데이터를 수정하지 못하도록 해야 한다. 
격리성은 동시성 성능 이슈와 직결되는 문제로 각 레벨을 설정 가능

4. 지속성(Durability) : 트랜잭션을 성공적으로 끝내면 무슨 일이 있어도 그 결과가 항상 데이터 베이스에 반영되야 한다. 예를 들어 중간에 애플리케이션 서버가 다운이 되는 등의 모종의 이유로
트랜잭션의 결과가 데이터 베이스에 반영이 되지 않을 경우, 데이터베이스 로그 등을 사용하든 어떻게 하든 성공한 트랜잭션 내용을 데이터 베이스에 반영해야 한다. 

 문제는 격리성이다. 트랜잭션 간의 격리성을 완벽하게 보장하기 위해서는 각 트랜잭션을 도착한 순서대로 그대로 실행해야 한다. 그러나 이건 매우 성능이 느리다. 
그래서 개발자가 격리성 4 레벨 중, 적정한 레벨을 판단해서 isolation 레벨을 선택하여 격리성을 보장함과 동시에 성능도 보장해야 한다.
