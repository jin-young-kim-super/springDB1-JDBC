개발자가 getConnection()을 통해 DB에 접근하려고 할 떄 아래와 같은 일이 발생한다. 
1. TCP/IP 연결
2. DB IP, username, password 전달
3. DB 내부에서 2의 정보를 바탕으로 인증 과정을 거치고 개발자만의 【DB 세션】 생성
4. 개발자에게 커넥션 반환
-> 그러나 만약 사용자가 동시에 100명이 오면, 동시에 위 과정을 100번 해야 하며, 이때마다 리소스 부하가 일어난다. 
이러한 문제점을 극복하고자 미리 몇 개의 커넥션을 만들어 놓고 요청될 떄마다 그냥 반환하는 커넥션 풀의 개념이 탄생한 것이다. 

커넥션 풀 숫자는 서비스의 특징와 애플리케이션 서버 스펙, DB 스펙에 따라 다르기 때문에 성능 테스트를 통해 최대 커넥션 풀 숫자를 설정해야 한다.
실무에서는 HICAKRI CP라는 커넥션 풀 오픈소스를 거의 대부분 쓴다.
스프링 부트 2.0부터는 HICARI CP를 디폴트로 등록해서 커넥션 풀을 사용한다.


DataSource 인터페이스(JDBC 표준 인터페이스)
이전 시간에 커넥션 풀을 배웠다.
1. DriverManager.getConnection() // 이건 커넥션 풀이 아닌, 요청 떄마다 DB와 연결을 맺어서 커넥션을 들고 오는 방식
2. Hicakri CP.getConnction()
3. 진영 CP.getConncetion()
-> 애플리케이션 로직에서 만약에 Hicakri CP를 사용하다가 "어? 그냥 고객 요청 없으니깐 DriverManager를 통해서 커넥션을 들고오자!!"라고 요구사항이 변경됐을 되면
Hicakari CP를 의존하고 있는 클라이언트 코드 부분을 DriverManger에 의존하게 코드 변경을 해야 한다. 이건 OCP 위반이다.
이러한 문제점을 해결하기 위하여 JDBC가 제공하는 DataSource 인터페이스가 있다.
이 인터페이스를 상속받아서 각 구현체(1,2,3)을 구현해 놓고, 클라이언트 코드에는 DataSource 인터페이스에만 의존하게 하면 OCP원칙을 해결할 수가 있다.
이러한 점을 위해 DataSource는 탄생을 하였다.
참고로 DrivaerManger를 까보면 알겠지만 이건 DataSource 인터페이스를 상속받지 않았다. 그래서 DataSource를 개발자가 직접 상속을 받아서 일일이 개발을 다시 해야 한다.
이런 문제점을 스프링은 DriverManagerSource라는 구현체를 제공한다. 이 구현체는 DataSource 인터페이스를 상속받아 구현하여 DriverManager를 사용할 수 있게 한다.
