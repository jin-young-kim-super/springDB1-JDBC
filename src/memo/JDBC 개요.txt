JDBC가 탄생하기 이전 과거에는 수많은 관계형 데이터 베이스마다 커넥션 하는 방법, SQL 전달하는 방법, 결과 응답을 받는 방법이 모두 달랐다. 
여기에서 2가지 문제점이 있다. 
1. 데이터베이스를 변경하면 애플리케이션 코드도 함꼐 변경해야 한다. 
2. 새로운 데이터 베이스를 다시 학습하여야 한다. 

그래서 JDBC 표준 인터페이스가 탄생했다. 
각 관계형 데이버 베이스 회사마다 예를 들어 MySQL 드라이버, Oracle 드라이버 등으로 구현체를 만들어, 사용자는 모든 관계형 데이터베이스를 같은 방법으로 다룰 수가 있게 됐다.
※JDBC의 한계 : SQL은 ANSI SQL이라고 표준이 있다. 그러나 각 벤더 회사는 이 표준을 지키지 않아서, 같은 기능이라도 SQL 문법이 조금씩 다르다.
그래서 JDBC를 사용하더라도 각 벤더 회사에 맞게 SQL은 거기에 맞게 작업을 해야 한다(참고로, JPA는 이런 점을 해결해 준다)


JDBC는 1997년도에 나온 오래된 기술이다. 그 사이에 JDBC를 편리하게 사용하는 기술들이 나왔다.
그 기술은 크게 2가지로 나뉜다.
1. SQL Mapper 기술(JDBC Template, MyBatis)
장점1 : SQL 응답을 객체로 자동 변환시켜 준다(옛날에는 자바 코드로 일일이 개발자가 매핑을 했다)
장점2 : JDBC의 반복 코드 제거(try-catch 안에서 getConnection을 일일이 해줬던 것을 기억해내라)
단점 : 개발자가 직접 SQL을 작성해야 한다
-> 이건 ORM이 SQL을 직접 작성하지 않는 것에 비교해서 생긴 단점

2. ORM 기술(JPA 등)
장점 : 개발자가 직접 SQL을 작정하지 않고, 예를 들어 insert(new Member())를 해주면 insert문을 자동으로 만들어 준다.
그래서 개발 생산성이 높아진다. 그리고 각 벤더마다 틀린 SQL문이 있어도 거기에 맞춰서 SQL문을 만들어 준다

단점 : 쉬운 기술이 아니므로 깊은 학습을 필요로 한다
