우리가 가장 흔한 패턴으로 [프레젠테이션 계층 -> 서비스 계층 -> 데이터 접근 계층]의 패턴을 가장 많이 사용한다. 
그럼 그렇게 사용하는 가장 큰 이유는 뭘까? 그건 바로 서비스 계층의 코드를 최대한 특정 기술(UI 기술, JPA 등)에 전혀 의존하지 않고 순수 자바 코드로만 작성하기 위함이 가장 큰 목적이다. 

프레젠테이션 계층은 UI와 관련된 기술이 웹, 서블릿, HTTP와 관련된 특정 기술을 가져간다. 
데이터 접근 계층은 특정 데이터 접근 기술을 가져간다.
이렇게 해서 서비스 계층에는 특정 기술의 코드가 없이, 순수 자바 코드로 유지할 수가 있는 것이다.  
※그래서 특정 기술이 변경이 되도 서비스 계층은 변경에 의한 영향을 최소한으로 할 수가 있다. 


순수 자바 코드 서비스 계층의 장점
1. 비지니스 로직을 유지보수하기가 쉽다.
-> 특정 기술 코드에 의존한 코드가 없고, 순수 자바 코드만 있으니 읽기도 쉽고 특정 기술 의식하지 않고 수정하기도 쉽다. 
 
2. 테스트 코드 짜기가 쉽다. 
-> Mock 코드다 짜기 쉽고, 유연한 테스트 코드도 짜기가 쉽다(만약 서비스 계층 의존관계에 특정 기술의 구현체가 있다면 Mock 코드도 못 짜고, 유연하게 의존관계를 변경도 못한다)

아래 코드를 분석해 보자.
트랜잭션은 서비스 계층에서 시작하는 것이 좋다. 그러나 그 결과 아래처럼
import javax.sql.DataSource;
import java.sql.Connection;
import java.sql.SQLException; 즉, JDBC라는 특정 데이터 베이이스 접근 기술에 의존을 하고 있다.
만약 JDBC에서 JPA 기술로 변경 시, 서비스 계층도 거기에 맞춰서 수정을 해야 한다. 즉 핵심 비지니스 로직과 특정 기술이 섞여져 있기에 읽기도 어렵고 특정 기술에 의존적인 자바 코드가 있기에
변경 시 유지/보수가 어렵다.（JDBC와 JPA의 트랜잭션 코드는 서로 다르다）
위에서도 언급을 했지만, 서비스 계층은 순수한 자바 코드로 만들기 위하여 데이터 베이스 관련 코드는 데이터 접근 계층이 가져가야 한다고 말했는데 이 코드는 거기에 맞춰져 있지가 않다.
그리고 참고로 SQLException등 데이터 관련 예외 또한 서비스 계층이 아니라 데이터 접근 계층에서 처리해줘야 한다.

지금까지 우리가 개발한 코드의 문제점
1. JDBC 코드 반복 문제

2. 데이터 접근 예외의 누수 문제
-> 데이터 접근 계층에 있어야 할 예외 코드(try-catch)가 서비스 계층에 있는 문제. 이상적으로는 데이터 관련 코드는 모두 데이터 접근 계층이 들고가야 한다.

3. 트랜잭션 커넥션 동기화 문제/트랜잭션 적용 반복 문제
->커넥션 동기화를 해결하기 위하여 커넥션을 파라미터로 넘겨 주었는데, 이를 위하여 update(), findById() 메서드를 커넥션을 파라미터로 받는 메서드와 받지 않는 메서드, 2개를 중복 구현해야 했다.
-> 트랜잭션 적용 반복 문제 : try-catch,finally가 계속 반복이 된다.
만약 트랜잭션 적용을 100번 해야 한다면 서비스 계층에 똑같은 코드가 100개 있어야 한다.

스프링은 순수 자바 코드의 서비스 계층 제공과 위 언급한 문제들을 해결하는 방법을 제공한다. 향후 학습 주제이다.

public class MemberServiceV2 {

    private final MemberRepositoryV2 memberRepository;
    private final DataSource dataSource;

    public void accountTransfer(String fromId, String toId, int money) throws SQLException {

        Connection connection = dataSource.getConnection();

        try {
            connection.setAutoCommit(false); // 트랜잭션 시작
            bizLogic(fromId, toId, money, connection);
            connection.commit(); // 트랜잭션 끝1
        }catch (Exception e) {
            connection.rollback(); // 트랜잭션 끝2
            throw new IllegalStateException(e);
        } finally {
            if(connection != null) {
                try {
                    // 커넥션 풀을 사용하지 않는다면 setAutoCommit(true)를 안 해도 된다.
                    // -> setAutoCommit(true)을 하지 않고 커넥션 풀에 반납을 하면 setAutoCommit(false)가 유지가 되 버린다.
                    // 그러나 커넥션 풀을 사용하지 않고 매회 커넥션 신규 생성 시 setAutoCommit은 false로 자동 세팅되게 DB에 설정이 되 있끼에
                    // 별 문제가 안된다.
                    connection.setAutoCommit(true);
                    connection.close();
                } catch (Exception e) {
                    log.error("error",e);
                    throw new IllegalStateException(e);
                }
            }
        }
    }

    private void bizLogic(String fromId, String toId, int money, Connection connection) throws SQLException {
        // 비니지스 로직 시작
        Member fromMember = memberRepository.findById(connection, fromId);
        Member toMember = memberRepository.findById(connection, toId);
        memberRepository.update(connection, fromId, fromMember.getMoney() - money);
        validate(toMember);
        memberRepository.update(connection, toId, toMember.getMoney() + money);
    }

    private static void validate(Member toMember) {
        if(toMember.getMemberId().equals("ex")) {
            throw new IllegalStateException("이체 중 예외 발생");
        }
    }
}

- 트랜잭션 추상화 -
트랜잭션 적용 시 jdbc와 jpa의 적용 코드가 다르다.
이떄 jdbc에서 JPA로 기술 변경 시 서비스 계층 코드도 변경이 된다는 문제가 있었다.
이에 대한 가장 단순한 해결법은 트랜잭션 인터페이스를 만들어서 각 기술에 대한 구현체를 따로 만들어 버리는 것이다.

아래와 같은 인터페이스를 만들어 각 기술마다 여기에 맞게 구현체를 제공하면 그만이다. 그러나 스프링은 이미 이런 인터페이스를 다 만들어 놓았고, 각 벤더들은 여기에 맞는 각 구현체를 만들어 놓았다.
public interface TxManager {
 begin();
 commit();
 rollback();
}
그래서 서비스 계층에는 이 인터페이스에 의존만 하면 데이터 접근 특정 기술이 변경이 되도 코드 변경이 없다.
스프링 트랜잭션 추상화 인터페이스인 PlatformTransactionManager 인터페이스을 제공한다.(주로 이걸 트랜잭션 매니저라고 줄여서 부름)

다음 시간에는 트랜잭션 커넥션 유지를 위한 파라미터 전달 방식을 스프링이 어떻게 해결하는 지 알아 보겠다.

- 트랜잭션 동기화 매니저 -
스프링은 트랜잭션 매니저와 별도로 트랜잭션 동기화 매니저를 제공한다.
트랜잭션 동기화 매니저를 통해 우리 코드의 문제점이 였던 트랜잭션의 커넥션을 파라미터로 일일이 넘겨줌으로 커넥션을 유지했던 문제점을 해결해 준다.
동작 방식
1. 트랜잭션 매니저는 트랜잭션 시작 시, 커넥션이 필요하므로 DataSource를 통해 커넥션을 만들고 만든 다음에 커넥션을 조회하여 트랜잭션을 시작한다.
2. 트랜잭션 매니저는 트랜잭션이 시작된 커넥션을 트랜잭션 동기화 매니저라는 곳에 보관이 된다(이떄 ThreadLocal에 보관이 되어서, 멀티쓰레드 환경에서도 안전하게 커넥션이 보관된다)
3. SQL문이 실행될 떄마다 트랜잭션 동기화 매니저에 보관된 같은 커넥션을 레포지토리 계층에서 꺼내어서 사용한다.(강의 자료 그림 참조)
4. 트랜잭션이 종료되면 트랜잭션 매니저는 트랜잭션 동기화 매니저에 있는 커넥션을 다시 꺼내서 그것을 통하여 트랜잭션을 종료시킨다.(commit/rollback)
※ThreadLocal에 대해서는 스프링 고급편에 다룸. 지금은 ThreadLocal은 해당 쓰레드 전용의 저장소라는 것만 알고 있자.(사실, 이러한 특성으로 인해 멀티쓰레드 상황에서도 커넥션이 안전하게 보관되는 것이다)
-> 트랜잭션 동기화 매니저로 인해 이제는 커넥션을 파라미터로 일일이 전달할 필요가 없어졌다.
